# PRD: 编辑器可用性修复与快捷键配置

**日期:** 2026-07-19
**状态:** Draft

## 1. 概述

### 1.1 问题陈述

Recordly 时间线编辑器存在三个阻碍高效剪辑的可用性问题：

1. **音频轨道无法切割**：按 X 键切割仅作用于视频轨道（`type="video"`），音频轨道（`type="audio"`、`type="audio_extra"`）被完全忽略，用户无法在播放头位置对音频片段进行分割。
2. **切割后右侧片段无法自由拖动**：视频片段切割后，右半段无法正常拖拽到其他时间位置。
3. **所有快捷键硬编码且不可配置**：时间线操作（X/S/I/O/←/→）和全局操作（Space/Ctrl+Z/Ctrl+Y）的快捷键均在代码中写死，设置页无快捷键管理入口，用户无法根据个人习惯自定义键位。

### 1.2 目标用户

使用 Recordly 进行屏幕录制后编辑的所有用户。

### 1.3 成功指标

- 在音频轨道中按 X 键，播放头所在的音频片段被切割为两段
- 视频片段切割后，右半段可被选中并拖拽到任意位置
- 设置页新增"快捷键"标签页，可查看、修改、恢复所有编辑器快捷键
- 修改后的快捷键重启应用后保留
- 快捷键冲突时提示用户，不允许保存

## 2. 功能需求

### 2.1 F1：音频轨道切割

#### F1.1 X 键切割扩展至音频轨道

- **范围**：Timeline widget 获得焦点时（`TimelineWidget.keyPressEvent`）
- **当前行为**：`_find_playhead_video()` 仅遍历 `track.type == "video"` 且 `clip.type == "video"` 的片段
- **期望行为**：
  - 检测播放头下方是否有任意可切割的片段覆盖（视频或音频）
  - 有 → 在播放头位置执行 `SplitClipCommand`，切割该片段为两段
  - 无 → 状态栏提示"播放头下无片段"
- **查找优先级**：
  - 按轨道索引升序遍历所有轨道（含 video / audio / audio_extra）
  - 在每条轨道内按 clip 索引升序查找播放头覆盖的片段
  - 返回第一个匹配的 `(track_index, clip_index)`，不区分类型
- **切割后选中行为**：切割后自动选中右半段（`clip_index + 1`）
- **约束**：
  - 切割操作可撤销/重做（复用现有 `SplitClipCommand`，无需修改命令层）
  - 播放头恰好在片段边缘时不执行切割（现有逻辑已处理）
  - 缩放轨道（`type="zoom"`）的片段不参与 X 键切割

#### F1.2 S 键切割保持现有行为

- S 键切割当前选中片段（无论类型），在播放头位置执行切割
- 如果选中的是音频片段，S 键应同样生效（已支持，验证无退化）

### 2.2 F2：切割后右半段拖动修复

#### F2.1 问题诊断

分割后 `SplitClipCommand` 在 `clip_index + 1` 位置插入右半段，`_selected_clip` 保持指向左半段（`clip_index`）。用户需手动点击右半段才能选中并拖拽。需排查的具体阻碍包括：
- 相邻两片段边缘紧贴时是否影响 `_hit_test` 识别
- `_drag_state` / `_drag_orig_*` 在拆分后的 segment 上是否正确初始化
- 拖动方向（正向/反向）是否存在非对称约束

#### F2.2 期望行为

- 切割后，无论左半段或右半段，均可被鼠标点击选中并拖拽至任意时间位置
- 右半段向左拖动时不被左半段阻挡（本轮不引入重叠检测）
- 右半段向右拖动时不被时间线边界异常截断
- 拖动操作可撤销/重做（复用现有 `MoveClipCommand`）

#### F2.3 验收标准

- [ ] 切割一段视频片段后，点击右半段可选中并高亮
- [ ] 选中右半段后向左拖拽，片段位置跟随鼠标实时更新
- [ ] 选中右半段后向右拖拽，片段位置跟随鼠标实时更新
- [ ] 释放鼠标后，`MoveClipCommand` 正确入栈，undo/redo 正常
- [ ] `source_start` / `source_end` 在拖动后与原始素材对齐关系正确

### 2.3 F3：可配置快捷键

#### F3.1 快捷键管理架构

新增 `core/shortcuts.py` 模块，定义：

```python
@dataclass
class ShortcutAction:
    action_id: str       # 唯一标识，如 "play_pause"
    display_name: str    # 设置页显示名，如 "播放/暂停"
    category: str        # 分类：播放控制 / 时间线编辑 / 全局
    default_keys: str    # 默认键位，如 "Space" / "X" / "Ctrl+Z"
```

所有快捷键注册集中管理，`TimelineWidget` 和 `MainWindow` 从该注册表读取当前绑定，不再硬编码。

#### F3.2 可配置的操作清单

| action_id | 显示名称 | 默认键位 | 分类 |
|-----------|---------|---------|------|
| `play_pause` | 播放/暂停 | Space | 播放控制 |
| `undo` | 撤销 | Ctrl+Z | 全局 |
| `redo` | 重做 | Ctrl+Shift+Z | 全局 |
| `redo_alt` | 重做（备用） | Ctrl+Y | 全局 |
| `split_at_playhead` | 在播放头处切割 | X | 时间线编辑 |
| `split_selected` | 切割选中片段 | S | 时间线编辑 |
| `delete_clip` | 删除选中片段 | Delete | 时间线编辑 |
| `delete_clip_alt` | 删除选中片段（备用） | Backspace | 时间线编辑 |
| `trim_in` | 裁剪入点 | I | 时间线编辑 |
| `trim_out` | 裁剪出点 | O | 时间线编辑 |
| `nudge_left` | 微移左移 0.5s | ← | 时间线编辑 |
| `nudge_right` | 微移右移 0.5s | → | 时间线编辑 |

#### F3.3 设置页快捷键标签页

在 `SettingsDialog` 中新增"快捷键"标签页（Tab），布局：

- **表格式列表**：每行一个操作，包含"操作名称 → 当前快捷键 → [编辑] [恢复默认]"
- **编辑交互**：点击 [编辑] 或双击快捷键单元格 → 弹出快捷键捕获对话框
  - 对话框标题："按下新快捷键"
  - 捕获用户下一次按键组合（含修饰键）
  - 显示当前捕获的键位描述（如 "Ctrl+Shift+K"）
  - "确定" → 校验冲突 → 保存 / "取消" → 放弃
- **冲突检测**：
  - 保存前检查是否与已有绑定冲突
  - 冲突时弹窗提示"与 [操作名称] 冲突，请重新设置"，不允许保存
  - 允许当前操作的旧绑定被覆盖（即先清除自身旧绑定）
- **恢复默认**：
  - 单行 [恢复默认] 按钮 → 该操作恢复为 `default_keys`
  - 标签页底部 [恢复全部默认] 按钮 → 所有操作恢复默认值，需二次确认

#### F3.4 持久化

- 快捷键映射通过 `QSettings("Recordly", "Recordly")` 持久化
- Key: `shortcuts/<action_id>` → Value: `"Space"` / `"Ctrl+Z"` 等字符串
- 应用启动时从 QSettings 读取并覆盖默认值
- 未配置的 action 使用 `default_keys`

#### F3.5 快捷键冲突时运行时行为

- 两个不同的操作绑定到同一快捷键时，两者都生效（Qt 正常分发）
- 冲突保护仅在设置页保存时生效：不允许将快捷键设为已被其他操作占用的绑定

#### F3.6 验收标准

- [ ] 设置页新增"快捷键"标签页，显示全部 12 个可配置操作
- [ ] 双击或点击编辑可弹出快捷键捕获对话框
- [ ] 捕获到的按键组合正确显示（含修饰键描述）
- [ ] 冲突快捷键保存时弹窗提示并阻止保存
- [ ] 单行恢复默认和全部恢复默认均正确工作
- [ ] 修改快捷键后关闭设置，对应操作立即使用新绑定
- [ ] 重启应用后修改的快捷键仍然生效
- [ ] X 键改为其他键位后，原 X 键不再触发切割

## 3. 用户故事

### US-1: 音频片段切割
**作为** 视频编辑用户
**我想要** 用 X 键在播放头处切割音频片段
**以便** 像编辑视频一样高效编辑音频轨道

**验收标准：**
- [ ] 播放头在音频片段上方时按 X，该片段在播放头处被切割为两段
- [ ] 切割后自动选中右半段
- [ ] X 切割音频结果与视频切割一致（`source_start` / `source_end` 正确计算）
- [ ] 切割操作可撤销/重做
- [ ] 播放头在 `audio_extra` 轨道的片段上方时同样生效

### US-2: 切割后片段自由拖动
**作为** 视频编辑用户
**我想要** 切割后左右两个半段都能自由拖动
**以便** 灵活调整片段在时间线中的位置

**验收标准：**
- [ ] 切割后点击右半段可选中并高亮
- [ ] 选中右半段后向左拖拽，片段正常左移
- [ ] 选中右半段后向右拖拽，片段正常右移
- [ ] 拖动操作可撤销/重做

### US-3: 自定义快捷键
**作为** 视频编辑用户
**我想要** 在设置中自定义所有编辑器的快捷键
**以便** 适配个人操作习惯，提高编辑效率

**验收标准：**
- [ ] 设置页可查看所有可配置快捷键及其当前绑定
- [ ] 可修改任意快捷键，保存后即时生效

## 4. Out of Scope

| 排除项 | 原因 |
|--------|------|
| 多选批量切割/移动 | 当前为单选模型，多选需改造选择系统（已在 `recordly-timeline-interaction-enhancements` 中排除） |
| 跨轨道片段拖动 | `MoveClipCommand` 已预留字段但 UI 无交互，应单独立项 |
| 拖动时片段重叠检测/阻止 | 本轮仅修复拖动功能，重叠检测增加复杂度需独立评估 |
| JKL 播放控制 | 属于播放控制范畴，已在之前 PRD 中排除 |
| 预设快捷键方案（Final Cut / Premiere 布局） | 当前用户量不足，优先支持手动自定义 |
| 快捷键导入/导出 | 本轮满足手动配置需求，跨设备同步可后续迭代 |
| 撤销/重做的历史面板 | 已在 `recordly-undo-redo-shortcuts` 中排除 |
| 缩放轨道片段参与 X 键切割 | 缩放片段切割语义不清晰，应独立设计 |
| macOS Cmd 键显式适配 | PyQt5 QKeySequence 自动处理 Cmd/Ctrl 映射 |

## 5. 技术约束

- 语言：Python 3.11+
- UI 框架：PyQt5
- 切割命令复用现有 `SplitClipCommand`，不修改命令层数据结构
- 快捷键持久化使用现有 `QSettings("Recordly", "Recordly")`
- 快捷键捕获对话框需过滤文本输入焦点（与 `_is_editor_active_and_safe` 逻辑一致）
- `_find_playhead_video()` 需重命名为 `_find_playhead_clip()` 并去除类型过滤
- 设置页从 `setFixedSize(520, 420)` 可能需要调整为更大尺寸以适应新标签页

## 6. 风险

| 风险 | 缓解 |
|------|------|
| 音频片段 `source_start`/`source_end` 计量单位与视频不一致（ms vs s） | SplitClipCommand 对两者使用相同 `speed` 倍率换算，audo clip 的 `speed` 为 1.0，计算应正确；实现前编写单元测试验证 |
| 快捷键注册从硬编码迁移到配置驱动时遗漏操作 | 以 action_id 为单元逐一迁移，每迁移一个运行现有测试确认无退化 |
| 切割后选中右半段可能破坏现有 trim_in/trim_out 的选中假设 | 仅在 `_split_clip` 内部设置 `_selected_clip = clip_index + 1`，trim 操作不受影响（trim 内先重新 split 再操作） |
