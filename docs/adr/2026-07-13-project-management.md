**影响范围**: 项目存储、项目管理 UI、缩略图

## ADR-001: 项目存储使用目录扫描而非 SQLite

### 上下文

Recordly 需要管理多个视频编辑项目。每个项目包含一个 JSON 元数据文件、一个缩略图、一个源视频文件。需要支持列出、创建、删除、重命名项目。

两种可选方案：
- **方案 A**: 使用 SQLite 数据库管理项目元数据，文件存储在别处
- **方案 B**: 目录扫描 — 每个项目一个子目录，通过扫描文件系统发现项目

### 决策

**选择方案 B：目录扫描。**

### 理由

1. **自包含性**：每个项目目录是完整独立的，用户可以直接复制、移动、备份到其他设备，无需导出/导入数据库
2. **零依赖**：不需要引入 SQLite 驱动或 ORM，减少依赖和打包体积
3. **人类可读**：用户可以随时在文件管理器中浏览项目文件，降低认知负担
4. **无迁移负担**：JSON 文件格式扩展只需加字段，不需要 ALTER TABLE 迁移脚本
5. **规模适配**：桌面录屏工具的典型项目数量在几十到几百个范围内，目录扫描的性能完全够用（< 10ms 扫描 100 个项目）

### 代价

- 不支持复杂查询（如按标签筛选）— 但当前需求不涉及
- 并发写入无保护 — 但桌面单用户单实例场景不需要
- 无法做增量更新 — 每次刷新全量扫描，但项目数量少，影响可忽略

### 备选方案评估

SQLite 方案在以下场景有优势，但当前不需要：
- 项目数量 > 1000
- 需要复杂筛选/排序/搜索
- 多用户并发访问

## ADR-002: 缩略图存储为侧车文件，Project JSON 引用路径

### 上下文

每个项目需要一张缩略图用于卡片展示。缩略图尺寸为 320×180 像素。两种存储方式：
- **方案 A**: 缩略图 base64 编码后嵌入 `project.json`
- **方案 B**: 缩略图存储为独立的 `thumbnail.png` 文件，`project.json` 中的 `thumbnail_path` 字段引用其路径

### 决策

**选择方案 B：侧车文件 + 路径引用。**

### 理由

1. **JSON 体积控制**：320×180 的 PNG 缩略图约 50-100KB，base64 编码后膨胀 33% 到 70-130KB，让 JSON 文件变得臃肿。每次加载项目列表时只读取元数据字段（name/duration 等），不需要解码缩略图
2. **独立可替换**：缩略图可以单独重新生成（如用户在时间线选了新帧），不需要修改 JSON 其他字段
3. **直接可用**：`thumbnail.png` 可以被文件管理器预览、图片查看器打开，不需要额外处理
4. **PyQt 友好**：`QPixmap` 可以直接从文件路径加载，比从 base64 解码+创建 QImage 更高效

### 代价

- 每个项目多一个文件 — 但文件数量少，无实际影响
- 缩略图文件可能丢失或被删除 — 通过 `thumbnail_path` 检查文件是否存在，丢失时使用默认占位图

### 备选方案评估

嵌入 base64 的唯一优势是"单文件即完整项目"。但 JSON 文件在文本编辑器中打开时，数万字符的 base64 字符串严重损害可读性。对于 Recordly 的用户场景，这个优势不成立。

## ADR-003: 项目列表使用卡片网格而非列表视图

### 上下文

需要展示项目列表，每个项目包含缩略图、名称、时长。两种 UI 形态：
- **方案 A**: 列表视图（`QListView`/`QListWidget`，每行一个项目）
- **方案 B**: 卡片网格（`FlowLayout` + 自定义 `QFrame`，缩略图在上，名称和时长在下）

### 决策

**选择方案 B：卡片网格。**

### 理由

1. **视觉识别效率**：缩略图是项目的主要识别特征。卡片网格将缩略图放在显眼位置（占卡片面积的 75%），用户可以快速通过视觉记忆找到目标项目，比文字列表逐行扫描高效得多
2. **空间利用率**：在 1280×800 的窗口尺寸下，卡片网格可以同时展示 4-5 列、约 12 张卡片，列表视图在同样高度下只能展示约 15 行但缩略图极小
3. **符合视频编辑工具的行业惯例**：Final Cut Pro、DaVinci Resolve、CapCut 等项目列表均使用网格/缩略图布局
4. **预留扩展空间**：卡片下方可以轻松添加更多元数据（如时长、标签、修改日期），而列表视图的列空间有限

### 代价

- 网格布局实现比列表视图复杂（需要 FlowLayout）
- 排序需要自定义逻辑（列表视图自带排序代理）

### 实现细节

使用自定义 `FlowLayout`（Qt 示例中的经典实现），将 `ProjectCard` 子类化 `QFrame`。网格列数根据容器宽度自动调整，卡片固定尺寸 240×180px。

### 备选方案评估

列表视图的优势在于实现简单、排序方便、适合大量文本型数据。但 Recordly 是视频编辑工具，缩略图是核心信息载体，列表视图会导致缩略图过小（通常 64×36px），失去实用价值。
