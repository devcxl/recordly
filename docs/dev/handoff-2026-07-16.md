# Handoff: 导出性能优化

## 上下文
- 项目: Recordly
- 当前阶段: Phase 1-3 已完成
- 分支: `feat/frame-parallel`（PR #66，含 Phase 1 + compose 并行 + 无 hwupload GPU）
- 基分支: `dev`

## 已完成的工作

### 三轮提交（PR #66 中）

1. **并行 resize+convert+tobytes** (`e493718`)
   - `_process_one()` + `_stream_frames_parallel()` 用 ThreadPoolExecutor
   - 有界窗口 futures，保持 ffmpeg 管道顺序写入

2. **compose 并行 + 无 hwupload_cuda** (`9e63a14`)
   - `Compositor.iter_frame_meta()` 新方法，返回原始帧引用
   - `_compose_and_encode` 在子线程中直接 compose
   - GPU 路径移除 `hwupload_cuda`，最终统一使用 rgb24 pipe
   - GPU 检测简化为 h264_nvenc 直连

3. **Phase 1 并发修复** (`e0f6636`)
   - loader 加 `threading.Lock` 保护 fh.seek/read + cache
   - CursorEffect 加 `_state_lock` 保护 smooth/trail/ripple
   - cache_size 12→256
   - 新增 `TestLoaderThreadSafety` + `TestCursorEffectThreadSafety`

### 诊断产出
- `docs/design/export-speed-optimization.md` — 分段 profiling + 四阶段方案
- `tools/profile_export.py` — 可复现 profiling 脚本

### 基准数据（16s 项目, 2560×1440→1920×1080）
- JPEG decode: 27.8ms/frame
- compose: 8.3ms (无 zoom) / 97.5ms (有 zoom)
- resize+convert+tobytes: 66.8ms/frame
- GPU 导出实测: 27.6s (35fps)

| 阶段 | ms/帧 | 占比 |

## 下一步

### Phase 2：zoom + resize 加速（目标 15-18s）
1. zoom crop resize `Image.LANCZOS→BILINEAR`（3-5× 加速）
2. compose 直接输出目标分辨率（跳过 exporter 的 2560→1920 resize）
3. compose 输出 RGB 而非 RGBA（跳过 convert("RGB")）

### Phase 3：预解码 + 乱序
4. 直接录制与重开路径使用 256 MiB decoded byte-LRU + inflight 去重
5. `FIRST_COMPLETED` 乱序准备，effects 与 FFmpeg 严格顺序执行
6. pending 按 256 MiB 单帧预算限制，禁止全量预解码
